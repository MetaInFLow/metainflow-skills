## 数据源优先级

| 信息类型 | 第一优先 | 第二优先 | 规则 |
|---|---|---|---|
| 企业工商、统一社会信用代码、注册资本、法人 | `metainflow-enterprise-query` | 客户档案表 / 历史 AP | 工商信息以企业查询结果为准 |
| 企业资质、证书、到期时间 | `metainflow-enterprise-query` | 历史 AP / 客户文档 | 同名资质按最新有效期更新 |
| 近期动态、融资、行业新闻、最新政策窗口 | `metainflow-web-search` | 最近纪要 / 旧 AP | 这类信息有时效性，写入前必须刷新 |
| 项目、商机、反馈、方案、合同 | `feishu-bitable` | 历史 AP | 业务金额类数据以多维表为准 |
| 决策链、跟进信息、竞品、风险 | 纪要 / 反馈 / 文档 | 历史 AP | 没证据不补写 |
| 知识库节点、已有 AP、客户目录文件 | `feishu_search_doc_wiki` + `feishu_wiki_space_node` + `feishu_drive_file` | - | 先搜索命中，再解析 wiki 节点和目录位置 |

## 章节与数据源映射

| 章节 | 主要字段 | 首选来源 | 更新动作 | 填写规则 |
|---|---|---|---|---|
| 一、企业概览 | 企业全称、信用代码、注册地址、法人、行业、员工规模、近期动态 | 企业查询 + 网页搜索 + 客户档案表 | `更新` | 基础工商字段可替换，近期动态应带时间语境 |
| 二、资质现状 | 资质名称、认定日期、有效期、状态 | 企业查询 + 历史文档 | `更新` | 同一资质只保留最新有效记录；状态仅写有效、即将到期、待确认 |
| 三、政策机会 | 优先级、政策名称、窗口、前置条件、匹配度 | 政策库索引表 + 企业信息 + 反馈 / 方案 | `更新` / `追加` | 只保留真正可行动的机会，优先级使用 `P0/P1/P2` |
| 四、决策链 | 角色、姓名、职务、态度、关注点、最近接触 | 拜访纪要、聊天、反馈、历史 AP | `更新` / `追加` | 角色固定为决策人、评估人、使用人、影响人；态度不明时写 `待补充` |
| 五、价值评估 | 已服务金额、在谈金额、潜在机会、续约概率、LTV | 项目总表 + 合同表 + 商机表 | `更新` | 金额必须可回溯；LTV 可为估算值，但要有依据 |
| 六、历史交互时间线 | 日期、事件、阶段、摘要、来源 | 纪要、反馈、方案表、合同表 | `追加` | 去重后按时间排序；来源字段必须保留 |
| 七、竞争情报 | 竞品、信息来源、竞争维度、优劣势 | 客户反馈、纪要、网页搜索 | `更新` / `追加` | 没有竞品证据时可省略具体内容，但保留表头 |
| 八、风险与预警 | 风险、等级、说明、应对建议 | 反馈表、合同表、纪要、商机表 | `更新` / `追加` | 风险等级仅写高、中、低；风险必须配应对建议 |
| 九、跟进策略 | 下一步行动、优先联系人、时间窗口、30/90天目标 | 上述全部证据综合分析 | `更新` | 先写最具体的行动，避免空泛表述 |
| 十、更新记录 | 日期、来源、摘要 | 本次执行上下文 | `追加` | 每次执行都新增一行，不覆盖旧记录 |

## 更新模式规则

### 1. 不覆盖历史

- 更新模式必须保留原有 AP 内容
- 时间线、更新记录默认 `追加`
- 政策机会、风险、决策链、竞争情报优先做局部更新，不整章重写

### 2. 去重规则

- 商机：按 `客户 + 政策名称/机会名称` 查重
- 时间线：按 `日期 + 事件 + 来源` 去重
- 资质：按 `资质名称` 去重，保留最新有效期
- 决策链：按 `姓名 + 职务` 去重，保留最新接触信息

### 3. 冲突处理

满足以下任一情况时，标记为 `待核实`，不要静默覆盖：
- 新旧企业名称、统一社会信用代码、法人信息不一致
- 不同来源给出不同预算、时间窗口或机会金额
- 决策人角色和态度判断互相冲突

### 4. 缺失值处理

- 不知道就写 `待补充`
- 不允许为了“看起来完整”而补数字、态度、预算或时间窗口

## 客户等级规则

| 等级 | 条件 | 跟进频率 |
|---|---|---|
| A | 在谈 + 潜在价值 `>= ¥20万`，或已签约年度客户 | 每周更新 AP |
| B | 在谈 + 潜在价值 `¥5万 - ¥20万`，且有明确需求 | 每两周更新 AP |
| C | 潜在价值 `< ¥5万`，或需求不明确 | 每月更新 AP |

## 回写字段清单

### 客户档案表

至少回写以下字段：
- 客户等级
- 客户标签
- 最近 AP 更新日期
- 机会汇总
- 关键风险摘要

### 商机表

新识别机会写入前必须先查重；若已有记录则更新以下字段：
- 机会名称
- 当前状态
- 预估价值
- 优先级
- 最近更新时间
- 来源（S28 / AP分析 / 手动）

## 常见错误处理

| 问题 | 处理方式 |
|---|---|
| 企业查询无结果 | 先改走模糊搜索，多候选时要求确认全称 |
| 客户档案表无记录 | 提示用户确认是否新建客户档案；不要直接写入错误客户 |
| 知识库无客户节点 | 用 `feishu_wiki_space_node.create` 建节点，或先用 `feishu-create-doc` 直接落到目标 wiki 节点 |
| 更新模式覆盖原内容 | 改为追加或局部替换，并在更新记录中标明来源和日期 |
| 商机重复写入 | 写入前先查重，已有记录只更新不新建 |
| 决策链证据不足 | 写 `待补充`，不凭空补人名或态度 |
| AP 长期未更新 | 依据客户等级提示更新周期，不把旧动态当最新状态 |
