# MetaInFlow Skills 收录规则

## 1. 仓库定位

本仓库用于统一收集 MetaInFlow 自研 Skills，并按适用工具与企业/场景进行组织。

## 2. 分支规则

- `main`：收录全部可公开维护的 Skills，作为全集主干。
- 企业/场景分支：按企业名或场景名建立，只保留适用于该企业或场景的 Skills。
- 企业/场景分支应基于 `main` 创建，再按适配范围做裁剪，不在分支内维护无关 Skills。
- 新增或更新 Skills 时，优先在 `main` 完成，再按需要同步到对应企业/场景分支。

## 3. 顶层目录规则

根目录按 Skill 主要适配工具分类，当前固定为：

- `OpenClaw/`
- `Coze/`
- `TUI-General/`

分类原则：

- 一个 Skill 只放在一个主分类目录下，避免重复维护。
- 若一个 Skill 可跨工具复用，按“主要运行平台”归类。
- 若确实需要跨目录复用，应通过文档说明适配关系，不直接复制多份。

## 4. Skill 收录规则

仅收录 MetaInFlow 自研、可维护、可复用的 Skills。每个 Skill 建议独立目录管理，例如：

```text
OpenClaw/
└── skill-name/
    ├── SKILL.md
    ├── README.md
    ├── assets/
    ├── templates/
    └── scripts/
```

最低要求：

- 必须有 `SKILL.md`，明确用途、触发条件、输入输出和使用方式。
- 目录名应使用清晰稳定的英文或英文短横线命名。
- 必须标注适用工具、适用场景和依赖前提。
- 如依赖脚本、模板或静态资源，应放在 Skill 自身目录下。

建议补充：

- `README.md`：补充背景、示例和限制说明。
- `examples/` 或 `templates/`：提供可复用样例。
- `CHANGELOG.md`：在迭代频繁时记录变更。

## 5. 收录判断标准

收录前至少满足以下条件：

- 有明确业务价值或场景复用价值。
- 指令边界清晰，不依赖过多隐式上下文。
- 使用步骤可被其他人重复执行。
- 依赖项明确，缺失依赖时有合理降级或说明。
- 不包含企业敏感信息、密钥、账号或不可分发内容。

以下内容不建议直接收录：

- 一次性临时脚本或无法复用的实验性说明。
- 与某企业强绑定且不适合复用，但又放在 `main` 中的内容。
- 缺少核心说明、无法独立运行或无法判断用途的半成品。

## 6. 企业/场景分支裁剪规则

企业/场景分支只保留：

- 该企业或场景确实需要的 Skills。
- 与该企业工具链、权限模型、交付方式兼容的 Skills。
- 已去除无关资产、示例和说明后的最小必要集合。

企业/场景分支不应保留：

- 明显不适用的工具目录或 Skill。
- 仅供其他客户使用的专用 Skill。
- 与当前场景冲突的默认配置或模板。

## 7. 命名与维护建议

- 企业/场景分支名建议直接使用企业名或场景名，例如 `BeeClaw`。
- Skill 名称应尽量稳定，避免频繁改名导致引用失效。
- 结构调整、归类变更或大版本升级时，应同步更新相关文档。

## 8. 提交前检查

提交前建议确认：

- Skill 已放入正确的顶层目录。
- `SKILL.md` 信息完整且可读。
- 无敏感信息、无无关文件、无重复副本。
- `main` 与企业/场景分支的边界清晰。
